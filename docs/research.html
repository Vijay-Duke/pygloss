English Over Python in the PyCharm Editor

How to render "englishish" text over Python source in the PyCharm editor — feasibility, the best rendering surfaces, how to generate the text cheaply, and a design informed by Google's Natural Language Outlines research and two rounds of adversarial review (codex CLI). Headline shifts: don't replace code line-by-line — layer a block-level outline over an editor that stays the source of truth; the audience is developers, all of it (PyCharm has no non-programmers), so this is a polyglot-dev reading aid, not an accountant tool; and there is no phased MVP — both reader profiles, all verbosity presets, all rendering surfaces, and a configurable LLM provider (OpenAI-compatible and Anthropic) ship on day one.

TL;DR

  • Feasible, ~1:1 port of a JetBrains folding plugin. PyCharm is the IntelliJ Platform; the rendering surfaces are language-neutral. Only two things change from a Java plugin: target Python PSI, depend on the Python plugin.
  • Don't lead with per-line "replace." Both Google's Natural Language Outlines research and codex review agree: per-line English is noise ("loop over invoices" adds nothing). Partition code into semantically-coherent blocks, one intent-carrying sentence each.
  • The editor stays the source of truth. English is an added, linked, disposable layer — not a replacement editor. Primary surface = a synced outline pane; secondary = block inlays; folding only for tiny deterministic idioms.
  • Generate deterministically first, let the LLM polish. Build the block structure + rough gloss from PSI (no hallucination, cacheable), then optionally have an LLM summarize. Local gpt-oss-20b via Ollama for privacy, or cheap cloud (Haiku 4.5).
  • Two settings, done right: verbosity as 3–4 named presets (Code / Hints / Outline / Reader), not a continuous slider; and swappable reader profiles ("Python for JS/Java/Go/C#" vs a general intent-summary aid) — all aimed at developers.
  • The audience is developers — there is no non-programmer in PyCharm. So "plain English for accountants" is a mirage. The real readers are all devs: polyglot devs reading Python, learners, people onboarding to an unfamiliar repo, and reviewers. Design for them.
  • No MVP — everything ships day one. Both reader profiles (polyglot lens + intent summary), all four verbosity presets, all three rendering surfaces, and full provider settings are in the first release. "Polyglot is deterministic, intent-summary is LLM-backed" is an implementation distinction, not a shipping order.
  • Configurable LLM provider, both ecosystems. Settings take model name + base URL + API key, and support both OpenAI-compatible and Anthropic endpoints — so local gpt-oss (Ollama), OpenAI, Groq/Together/OpenRouter, and Claude all work behind one setting.
  • Permanently excluded (safety/scope, not phasing): per-line replacement, bidirectional sync (edit English → rewrite Python), a full custom editor, LSP, and a continuous slider.

The core mechanism

Three platform surfaces let you render text over/around code without touching the file on disk. They compose.

A. Code folding (FoldingBuilderEx) — the "replace" surface

buildFoldRegions() walks the PSI and emits fold regions; getPlaceholderText() returns your English; isCollapsedByDefault()=true shows English first. Registered via com.intellij.lang.foldingBuilder. Expanding restores the real Python; the file never changes. Ceiling: placeholder text is single-line, unstyled, and folding is synchronous — so it fits rule-based text only, never async LLM prose. Use it only for tiny idioms (comprehensions, boolean expressions, obvious if/return).

B. Inlay hints / Code Vision — the "overlay" surface

Renders extra text around code without hiding it. DeclarativeInlayHintsProvider (2023.1+) computes in the background, so it tolerates async/LLM text. For richer block UI than declarative inlays allow, drop to InlayModel / EditorCustomElementRenderer, or CodeVisionProvider / DaemonBoundCodeVisionProvider for a prose line above each declaration (PSI-bound → auto-invalidates on edit). This is the in-editor "overlay" look closest to Advanced Java Folding.

C. Rendered / reader mode — precedent for a toggled English view

JetBrains editors already render non-code in place: Javadoc/Markdown reader mode, toggled by a gutter icon (Ctrl+Alt+Q, "Render documentation comments in place"). This is proof the platform supports a toggled, rendered-in-editor view — the natural home for a "show English" toggle.

Rejected surfaces: a full custom editor (fights PyCharm's editing, inspections, navigation, refactoring, VCS, debugger, accessibility) and LSP (this is a PSI projection problem, not a language-server one). JetBrains MPS is the conceptual north-star — a projectional editor where English would be one notation of the same AST — but MPS is a separate workbench with no Python, so it's inspiration, not a usable path.

PyCharm / Python specifics

Declare the Python plugin as an optional dependency so one build serves Community + Pro:

<!-- plugin.xml -->
<depends optional="true" config-file="python-features.xml">PythonCore</depends>  <!-- PyCharm Community -->
<depends optional="true" config-file="python-features.xml">Pythonid</depends>    <!-- PyCharm Professional -->
<!-- or module-style: -->
<depends optional="true">com.intellij.modules.python</depends>

Gradle (IntelliJ Platform Gradle Plugin 2.x): add bundledPlugin("PythonCore") (or Pythonid). Target Python PSI in com.jetbrains.python.psi: PyFunction, PyClass, PyIfStatement, PyForStatement, PyWithStatement, PyTryExceptStatement, PyCallExpression, PyComprehensionElement, PyDecorator, PyYieldExpression, etc. Everything else (extension points, folding/inlay APIs, threading, read actions) is identical to a Java plugin — the rendering layer is language-neutral.

Generating the English

Rendering is solved; what English to show is the product. The winning shape, per the research and review:

Deterministic skeleton first, LLM polish second

Build the block structure and a rough gloss from PSI — deterministic, offline, cacheable, debuggable, zero hallucination. Only then hand a block to an LLM to summarize/refine. Don't let the LLM own the whole job; it multiplies hallucination and cost.

  • Rule-based (skeleton): templates per PSI node. for x in items: → block "for each item…"; comprehension → "the list of … for every … where …". Instant, deterministic.
  • LLM (polish / whole-function intent): "keep the invoices over the threshold that aren't flagged, mark each approved, and return them." Async only.

Model & cost

  • Cheapest cloud: Claude Haiku 4.5 — $1 / $5 per 1M tokens (input/output). A short function ≈ 150 in / 80 out ≈ $0.0006/call uncached.
  • Local / private: gpt-oss-20b via Ollama (ollama run gpt-oss:20b, OpenAI-compatible at localhost:11434). Source never leaves the machine — the decisive factor for a code tool. gpt-oss-120b needs ~80GB VRAM and is overkill; the 20B is plenty for englishish. Make the endpoint pluggable — both speak OpenAI-compatible JSON.
  • Two-layer cache (the real cost lever):
  • On-disk cache keyed by a normalized PSI-subtree hash — skip the call entirely when code is unchanged. In an editor this kills the vast majority of calls.
  • Provider prompt caching for the calls that do fire — cache the stable system prompt (rules + few-shot); the code snippet is the volatile suffix. Haiku's minimum cacheable prefix is 4096 tokens.
  • Full cache key: {psiHash, audienceProfile, verbosityLevel, promptVersion, pluginVersion}.

Architecture constraint

Folding is synchronous → rule-based text only. Any LLM/network text flows through async inlays / code-vision / the outline pane, never the folding builder. Render "…" until the async result resolves, then swap.

Model & provider settings

The LLM-backed features (intent summaries, and optional polish on the polyglot lens) must work against whatever model the user has. Settings expose three fields plus a provider switch:

FieldPurposeExample
ProviderWhich wire protocol to speakOpenAI-compatible \Anthropic
Base URLEndpointhttp://localhost:11434/v1 (Ollama) · https://api.openai.com/v1 · https://api.anthropic.com
Model nameFree-text model idgpt-oss:20b · gpt-4o-mini · claude-haiku-4-5
API keyAuth (blank for local)stored in the IDE's PasswordSafe, never in plain settings XML

Two adapters behind one interface — the protocols differ and can't share a client:

  • OpenAI-compatible → POST {baseUrl}/chat/completions, Authorization: Bearer , messages array, response at choices[0].message.content. One adapter covers OpenAI, Ollama, Groq, Together, OpenRouter, LM Studio, vLLM — anything that speaks the OpenAI shape (incl. local gpt-oss).
  • Anthropic → POST {baseUrl}/v1/messages, x-api-key: + anthropic-version header, system sent separately, max_tokens required, response at content[0].text, model ids like claude-haiku-4-5.

The Provider switch selects the adapter; Base URL / Model / Key are free-text so any current or future model works without a plugin update. The deterministic polyglot path needs no provider at all — it's a local lookup table — so the plugin is fully functional with zero API config; the provider only powers the LLM summaries. Ship sensible defaults (e.g. OpenAI-compatible → localhost:11434/v1 → gpt-oss:20b) so a local Ollama user has nothing to configure.

The two settings

Setting 1 — verbosity as named presets (not a continuous slider)

A continuous "% English" slider is unpredictable — nobody knows what 40% means. Use discrete presets that each choose an experience; internally each adjusts granularity depth, prose detail, and which surfaces are active:

PresetSurfaces onGranularityProse
Codefolding only (or nothing)—none
Hintsblock inlaysfunction / blockterse one-liners
Outlineoutline pane + key inlaysfile → class → function → blocksummary sentences
Readerfull rendered viewdown to block/line on demandfuller narrative

Setting 2 — audience profiles (the bigger idea)

Everyone here is a developer (PyCharm has no non-programmers), but developers read Python for different reasons. Model each reason as a swappable profile over the same engine:

  • Intent summary (learners, onboarding to an unfamiliar/legacy repo, fast review) — block-level "what does this do" prose, LLM-heavy, higher hallucination/trust risk. Not an accountant tool — a reading aid for devs facing unfamiliar code.
  • Python for JS / Java / Go / C# (polyglot developers reading Python) — explain Python-specific concepts (async/await semantics, with context managers, comprehensions, decorators, self, *args/kwargs, yield/generators, the GIL, dunder methods, :=) and show the reader's-language analogy where one exists. Mostly a deterministic concept → {explanation, per-language analogy}** lookup table keyed by PSI node/pattern — low hallucination, no per-code LLM needed, concept-triggered so sparse.

Critical wording rule: never say "Equivalent." Python async def ≠ JS async function (different concurrency model); with ≈ but isn't Java try-with-resources / C# using; a decorator is not a Java annotation; yield is not "return multiple values"; self is explicit binding, not this. Label every mapping "Closest analogy / Compare to", add a one-line lossy caveat, and attach a confidence tier. Sparse, deterministic, honest callouts read as trustworthy; chatty magic reads as a liar.

The two profiles

Both ship day one; they differ in how the English is produced, which drives their risk profile.

Audience reality anchors both. PyCharm is a developer IDE — non-programmers never open it, so "plain English for accountants" targets a user who isn't there. Every real reader is a developer; the useful splits are all dev-shaped: polyglot devs reading Python, learners, people onboarding to unfamiliar/legacy code, and reviewers.

codex (two review rounds) rated the polyglot lens the more defensible, lower-risk half — worth leaning the marketing and QA weight toward it even though both ship together:

  • The polyglot lens is a deterministic concept→analogy table. Polyglot devs already know control flow, types, refactoring; they need idiom translation, not magical comprehension — low hallucination, sparse, table-driven. Credible wedge: "I know JS/Java/Go — help me read Python idioms in this repo."
  • The intent-summary profile is LLM-backed and carries the real risk: a general "explain what this does" promise pushes toward false confidence and "impressive on small examples, brittle in real code." Keep it honest with confidence tiers, stale badges, and the deterministic PSI skeleton underneath.

One product, swappable profiles, both present at launch.

Prior art & sources

  • Google — Natural Language Outlines for Code (2024): block-level prose outlines interleaved literate-programming style; bidirectional sync; 90% of outlines rated acceptable+. The core "not per-line, use blocks" evidence.
  • Advanced Java Folding 2 (Fork): the reference plugin — folding-based syntax rewrite in-editor, file unchanged. The mechanism to port.
  • JetBrains MPS — projectional editing / diverse notations: the north-star concept (English as a notation of the AST); separate workbench, no Python.
  • IntelliJ Reader Mode / render docs in place: precedent for a toggled rendered-in-editor view.
  • IntelliJ SDK — Folding Builder
  • IntelliJ SDK — Inlay Hints
  • IntelliJ SDK — PyCharm Plugin Development
  • InlayModel (richer editor inlays)

Permanently excluded

These stay out on scope/safety grounds regardless of the "everything day one" rule:

  • Per-line replacement — noise; use block-level outlines.
  • Bidirectional sync (edit English → LLM rewrites Python) — footgun for a reading tool: false authority, weak auditability, silent wrong rewrites. If it's ever wanted, it must be an explicit assisted-edit → propose patch → show diff → require approval → run tests. Never silent, and out of the initial scope by design.
  • Full custom editor / LSP — fights the platform; unnecessary for a projection.
  • Continuous verbosity slider — named presets instead.
  • The word "Equivalent" — use "Closest analogy" with caveats.

Day-1 scope

No phased MVP. The first release contains the full feature set:

  1. Engine: Gradle IntelliJ Platform 2.x, bundledPlugin("PythonCore"), optional depends; PSI block detector + deterministic skeleton + normalized-PSI-hash cache ({psiHash, profile, level, promptVersion, pluginVersion}).
  2. Polyglot lens — concept→analogy table for async/with/comprehension/decorator/self/*args/yield/:=/dunder, with "Closest analogy" + confidence tier + lossy caveat. Target languages: JS, Java, Go, C#.
  3. Intent-summary profile — LLM block/function summaries via async inlays + outline pane, deterministic skeleton underneath, stale badge + regenerate.
  4. All three rendering surfaces — synced outline pane (primary), block inlays / Code Vision (overlay), folding (tiny idioms).
  5. Verbosity presets — Code / Hints / Outline / Reader.
  6. Provider settings — Provider (OpenAI-compatible | Anthropic) + Base URL + Model name + API key (in PasswordSafe); defaults to local Ollama gpt-oss:20b.
  7. Affordances — quick-doc hover, gutter icons (explain / cache / stale), export outline.

Top risks: subtly-wrong summary (trust); stale cache after edits (→ stale badge + regenerate); inlay noise; LLM latency/flicker; and provider/Ollama setup becoming the product (mitigated — the polyglot lens needs no provider, so the plugin works with zero config). The old "audience mismatch" risk is designed out — the plugin targets developers, PyCharm's only users.
